Actualización cada 5 segundos

Definido en LocationService.kt como LOCATION_INTERVAL = 5000L. El mismo valor controla tanto la solicitud GPS como el envío al servidor — no hay separación entre frecuencia de muestreo y frecuencia de reporte.

Flujo paso a paso
1

📡 GPS / FusedLocationProviderClient

DefaultLocationProvider construye un LocationRequest con PRIORITY_HIGH_ACCURACY y lo registra en FusedLocationProviderClient. Valida permisos y que GPS + Network estén habilitados antes de iniciar. El resultado llega como un Flow<Location> via callbackFlow.

⚙️ LocationService (Foreground Service)

Escucha el flow con .onEach { location → }. Por cada emisión: actualiza la notificación persistente, escribe en el archivo local android_location (log de respaldo), y llama al use case UpdateLocation. Corre en Dispatchers.IO con un SupervisorJob.

3

🗄️ LocationRepositoryImpl

Construye el payload enriqueciendo la coordenada con: vehicleCode (leído desde Room/cache de operación), androidId del dispositivo, timestamp actual con TimeUtils, e idIncident si hay un incidente activo en cache. Todos estos datos son lecturas síncronas del cache local.

4

🌐 HTTP POST → API de ubicación

LocationRemoteDataSource llama a LocationApi.updateLocation() via Retrofit, envuelto en NetworkApi.apiCall{}. El endpoint es LOCATION_URL + "location" (varía por build variant). Si el servidor responde con error, apiCall lo mapea a un BannerModel — pero el resultado es ignorado en LocationService (fire-and-forget).

Puntos de falla potenciales
🔥
Fire-and-forget sin reintentos
Alta severidad

El resultado de UpdateLocation se llama con invoke() pero su Result<Unit> es ignorado completamente en LocationService. Si la red falla, el dato se pierde sin posibilidad de reintento.

LocationService.kt:88 — updateLocation.invoke(lat, lon) // resultado descartado
📶
Sin offline queue / persistencia
Alta severidad

Si el dispositivo pierde conectividad (túneles, zonas rurales), los envíos fallidos simplemente se descartan. El archivo local android_location acumula datos pero nunca se usa para reenvío automático.

LocationService.kt:96 — storeLocation() escribe pero nunca relee
🧵
Callback en MainLooper
Alta severidad

FusedLocationProviderClient.requestLocationUpdates() recibe Looper.getMainLooper(). Si el hilo principal está ocupado, los callbacks de ubicación se retrasan o se pierden.

DefaultLocationProvider.kt:64 — Looper.getMainLooper()
🚗
vehicleCode vacío en el payload
Media severidad

Si la cache de operación aún no cargó (app recién iniciada, error previo), vehicleCode llega como "" gracias al .orEmpty(). El servidor podría rechazar o guardar datos inválidos.

LocationRepositoryImpl.kt:27 — .vehicleCode.orEmpty()
🔒
Detención silenciosa del servicio
Media severidad

Android puede matar el foreground service en condiciones de memoria baja. El flag de retorno super.onStartCommand() por defecto no garantiza reinicio automático (START_STICKY no está explícito).

LocationService.kt:56 — return super.onStartCommand(...)
⚡
Batería: HIGH_ACCURACY siempre activo
Media severidad

PRIORITY_HIGH_ACCURACY activa GPS, WiFi y red celular continuamente. En zonas con mala señal GPS el consumo se dispara y el dispositivo puede entrar en modo ahorro de batería, interrumpiendo el servicio.

DefaultLocationProvider.kt:50 — Priority.PRIORITY_HIGH_ACCURACY
📏
minDistance 50m ignora movimientos menores
Baja severidad

Si la ambulancia está estacionada o en tráfico lento (<50m en 5s), FusedLocation puede no emitir nuevas coordenadas aunque hayan pasado los 5 segundos. El servidor no recibe heartbeat de "siguen en el mismo lugar".

DefaultLocationProvider.kt:49 — setMinUpdateDistanceMeters(50f)
🕐
Sin separación muestra/envío
Baja severidad

El intervalo GPS y el intervalo de envío HTTP son el mismo valor hardcodeado. No es posible ajustar "muestreo rápido + envío cada N muestras" sin tocar el servicio.

LocationService.kt:30 — LOCATION_INTERVAL = 5000L
Mejoras recomendadas
⭐ Recomendado — Mayor impacto

Cola de envío con WorkManager

Encolar cada actualización de ubicación como una OneTimeWorkRequest con NETWORK_CONNECTED como constraint. WorkManager garantiza el reintento automático con backoff exponencial, survives reboots, y libera al servicio de manejar fallos de red.

  • Cero pérdidas por corte de red
  • Reenvío automático al recuperar conectividad
  • Backoff configurable (lineal o exponencial)
  • Persiste entre reinicios del dispositivo
// LocationWorker.kt val request = OneTimeWorkRequestBuilder<LocationWorker>() .setConstraints( Constraints(requiredNetworkType = CONNECTED) ) .setBackoffCriteria(EXPONENTIAL, 15, SECONDS) .setInputData(workDataOf( "lat" to lat, "lon" to lon )) .build() WorkManager.getInstance(context).enqueue(request)
⭐ Recomendado — Confiabilidad

START_STICKY + reinicio explícito

Devolver START_STICKY en onStartCommand para que Android reinicie el servicio automáticamente si es matado por el sistema. Complementar con un BroadcastReceiver para BOOT_COMPLETED.

  • El servicio se reactiva tras muerte por OOM
  • Arranque automático post-reinicio del dispositivo
  • Sin intervención del usuario
override fun onStartCommand(...): Int { when (intent?.action) { ACTION_START -> start() ACTION_STOP -> stop() } return START_STICKY // ← garantiza reinicio }
✦ Optimización

Looper dedicado para GPS

Mover los callbacks de FusedLocationProviderClient a un HandlerThread dedicado en lugar del MainLooper, para evitar contención con el hilo de UI.

  • El UI thread queda libre para renderizado
  • Callbacks GPS sin retrasos por jank de UI
  • Cambio de 2 líneas en DefaultLocationProvider
val handlerThread = HandlerThread("LocationCb") .also { it.start() } client.requestLocationUpdates( request, locationCallback, handlerThread.looper // ← no más MainLooper )
✦ Optimización

Prioridad adaptativa según velocidad

Usar PRIORITY_BALANCED_POWER_ACCURACY cuando el vehículo está detenido y cambiar a HIGH_ACCURACY solo cuando hay movimiento significativo. Reduce consumo de batería hasta un 40% en escenarios de espera.

  • Menor consumo en standby
  • Full accuracy en movimiento
  • Extiende vida de batería en turnos largos
◈ Mejora adicional

Heartbeat periódico aunque no haya movimiento

Enviar un "sigo aquí" al servidor cada N segundos aunque la posición no haya cambiado los 50m mínimos. Permite al backend distinguir "ambulancia detenida" de "app caída".

  • Mejor observabilidad en el backend
  • Detección de dispositivos offline
  • Implementable con un timer paralelo en el servicio
◈ Mejora adicional

Aprovechar el archivo local para reenvío

El archivo android_location ya existe y acumula datos. Agregar un job periódico (WorkManager PeriodicWorkRequest) que lea entradas sin confirmar y las reenvíe, marcándolas como enviadas.

  • Reutilizar infraestructura ya existente
  • Recuperación de datos en gaps de conectividad largos
  • Log de auditoría con timestamps exactos
