Flujo de recepción paso a paso

🔥 Firebase Cloud Messaging (FCM)

El backend envía un mensaje FCM tipo data payload (no notification payload). Llega como Map<String, String> con un campo type que determina el tipo de notificación. FCM gestiona la entrega en background a nivel de sistema operativo.

📄 google-services.json · project: sisem-notification

⚙️ MessagingService.onMessageReceived()

FirebaseMessagingService recibe el mensaje. Si remoteMessage.data.isNotEmpty() llama a handlePushNotification(). Usa un CoroutineScope(Dispatchers.Main + SupervisorJob) — las coroutines se ejecutan en el hilo principal.

📄 MessagingService.kt · línea 26
3

🔔 NotificationsManager.buildNotificationData()

Parsea el Map<String, String> a un NotificationData según el tipo. Luego hace dos cosas en paralelo:

Notificación Android

Crea NotificationCompat con sonido, prioridad HIGH, y PendingIntent a MainActivity. La publica con NotificationManager.notify().

EventBus

Publica el NotificationData en el MutableSharedFlow del EventBus para mostrar el popup en UI inmediatamente.

🗄️ NotificationRepositoryImpl.storeNotification()

Persiste en Room y ejecuta lógica específica según el tipo:

INCIDENT_ASSIGNED

Fetch remoto de detalles del incidente → extrae coordenadas de geolocation → actualiza cache del incidente.

UPDATE_VEHICLE_STATUS

Refetch remoto de operationConfig → actualiza cache de operación.

Puntos de falla — No recepción de notificaciones
🔕
POST_NOTIFICATIONS no concedido (Android 13+)
Alta severidad

En Android 13+, POST_NOTIFICATIONS es un permiso de runtime. Si el usuario lo deniega, las notificaciones del sistema no se muestran. El app no verifica si el permiso fue concedido antes de intentar mostrar notificaciones, y no tiene flujo de re-solicitud.

AndroidManifest.xml — POST_NOTIFICATIONS declarado pero sin verificación en runtime
🔑
Token FCM no registrado en el backend
Alta severidad

El token FCM se loguea en SisemApplication pero nunca se envía al servidor. No existe endpoint ni lógica para registrar/actualizar el token del dispositivo. Si el backend no tiene el token actualizado, ninguna notificación llega.

SisemApplication.kt — Timber.d("FCM token: ${task.result}") // solo log, nunca se envía
🔄
Token FCM rotado sin actualización en backend
Alta severidad

onNewToken() se llama cuando FCM rota el token (reinstalación, borrado de datos, cambio de cuenta Google). El método solo loguea el nuevo token — nunca lo envía al backend. El dispositivo queda invisible para FCM.

MessagingService.kt:34 — onNewToken(token) { Timber.d(...) } // token perdido
📡
INCIDENT_ASSIGNED: fetch remoto puede fallar silenciosamente
Alta severidad

Al recibir INCIDENT_ASSIGNED, se hace un fetch remoto de los detalles. Si falla (sin red, timeout), el incidente se guarda en Room con datos parciales (solo los del payload FCM, sin prioridad ni detalles). El operador ve una notificación pero el mapa puede quedar sin datos.

NotificationRepositoryImpl.kt:56 — .onFailure { IncidentEventHandler.publishIncidentErrorEvent(it) }
💣
checkNotNull() crashea en IPS y Transmi
Alta severidad

handleIpsPatientTransferredNotification() y handleTransmiNotification() usan checkNotNull() sin manejo de excepción. Si no hay incidente activo en cache (ej: app recién iniciada, cache limpiada), lanza IllegalStateException que mata la coroutine silenciosamente.

NotificationRepositoryImpl.kt:72 — checkNotNull(notification.geolocation?.split(","))
⚡
EventBus con collectLatest — notificaciones simultáneas se descartan
Media severidad

El EventBus usa collectLatest, que cancela el procesamiento anterior cuando llega uno nuevo. Si dos notificaciones llegan en ráfaga (ej: incidente + apoyo en camino), solo la segunda se procesa en UI.

EventBus.kt:14 — events.filterIsInstance<T>().collectLatest { onEvent(it) }
🔒
PendingIntent con FLAG_ONE_SHOT — solo funciona una vez
Media severidad

El PendingIntent de la notificación del sistema usa FLAG_ONE_SHOT. Si el usuario recibe dos notificaciones del mismo tipo y toca la segunda sin haber tocado la primera, el intent puede ya no ser válido y no se navega correctamente.

NotificationsManager.kt:68 — FLAG_ONE_SHOT or FLAG_IMMUTABLE
📦
Índice único en data puede rechazar duplicados válidos
Media severidad

La tabla notification tiene un índice único sobre el campo data. Si el backend reenvía la misma notificación (retry de FCM), Room la ignora con REPLACE — no es un crash, pero sí puede provocar que notificaciones reenviadas no se muestren al usuario.

NotificationEntity.kt — Index(value = ["data"], unique = true)
🏃
Dispatchers.Main en MessagingService — riesgo de ANR
Media severidad

El serviceScope usa Dispatchers.Main. Si el procesamiento de la notificación (fetch remoto + escritura Room) tarda, bloquea el hilo principal del servicio. En Android el FCM service tiene un timeout limitado (~10s) antes de ser terminado por el sistema.

MessagingService.kt:19 — CoroutineScope(Dispatchers.Main + SupervisorJob())
⏱️
delay(500) hardcodeado en MainActivity
Baja severidad

Al abrir el app desde una notificación, hay un delay(500) antes de publicar el evento en el EventBus. Si la UI no está lista en ese momento (pantallas pesadas), el popup puede perderse.

MainActivity.kt — delay(500) // workaround sin garantía
🌐
Sin canal de notificación diferenciado por tipo
Baja severidad

Todas las notificaciones van al mismo canal "notifications" con importancia DEFAULT pero prioridad HIGH. El usuario no puede silenciar solo los tipos menos críticos (AUTH_TIME) sin silenciar también los urgentes (INCIDENT_ASSIGNED).

NotificationsManager.kt — CHANNEL_ID = "notifications" (único para todos)
Oportunidades de mejora
⭐ Crítico — Implementar ahora

Registrar el token FCM en el backend

Sin esto, el backend no sabe a qué dispositivo enviar. Enviar el token en login y en onNewToken(), asociándolo al usuario/vehículo activo.

  • Registrar en login exitoso
  • Actualizar en onNewToken()
  • Limpiar al hacer logout
// MessagingService.kt override fun onNewToken(token: String) { serviceScope.launch { registerTokenUseCase.invoke(token) // ← enviar al backend } }
⭐ Crítico — Confiabilidad

Verificar permiso POST_NOTIFICATIONS en runtime

Mostrar un diálogo explicativo antes de solicitar el permiso y ofrecer ir a Settings si fue denegado permanentemente.

  • Verificar al iniciar sesión
  • Banner persistente si está denegado
  • Deep link a Settings del sistema
if (!hasNotificationPermission()) { requestPermissionLauncher.launch( Manifest.permission.POST_NOTIFICATIONS ) }
⭐ Recomendado

Reemplazar collectLatest por collect en EventBus

Cambiar a collect para garantizar que todas las notificaciones se procesen, incluso en ráfagas. Si el popup es costoso, usar un buffer o una cola.

  • Ninguna notificación se descarta
  • Ráfagas de notificaciones manejadas correctamente
  • Cambio de 1 línea
// EventBus.kt events.filterIsInstance<T>() .collect { event -> // ← no más collectLatest coroutineContext.ensureActive() onEvent(event) }
✦ Optimización

Mover MessagingService a Dispatchers.IO

El procesamiento de notificaciones (fetch + Room) debe correr en IO, no en Main. El hilo principal debe quedar libre para la UI.

  • Sin riesgo de ANR en el servicio FCM
  • Mayor tiempo disponible para procesar antes del timeout del sistema
// MessagingService.kt private val serviceScope = CoroutineScope(Dispatchers.IO + serviceJob)
✦ Optimización

Canales diferenciados por criticidad

Crear al menos 2 canales: uno para alertas operativas críticas (INCIDENT_ASSIGNED) y otro para informativas (AUTH_TIME, UPDATE_VEHICLE_STATUS).

  • Canal URGENTE: importancia HIGH, sonido específico
  • Canal INFORMATIVO: importancia DEFAULT, sin sonido
  • El usuario puede configurarlos individualmente
✦ Seguridad

Proteger checkNotNull con manejo de errores

Reemplazar los checkNotNull() sin catch en IPS y Transmi por validaciones explícitas que logueen y retornen sin crash.

  • Evita crasheos silenciosos en coroutines
  • Logueo de Timber para debugging
  • Comportamiento degradado en lugar de fallo total
val incident = incidentCacheDataSource .observeActiveIncident().first() ?: return Timber.w("No active incident")
◈ Mejora adicional

Reemplazar delay(500) con lifecycle-aware

Usar repeatOnLifecycle(RESUMED) para publicar el evento solo cuando la UI esté lista, eliminando el delay hardcodeado.

  • Sin race condition entre UI init y eventos
  • Funciona con cualquier velocidad de carga
◈ Mejora adicional

Aprovechar el payload FCM para INCIDENT_ASSIGNED

El payload ya trae incidentType, incidentPriority, address, geolocation. Mostrar el incidente inmediatamente desde el payload sin esperar el fetch remoto. Hacer el fetch en segundo plano para enriquecer.

  • Incidente visible aunque falle la red
  • UX más rápida — sin spinner esperando el fetch
  • Fetch remoto como enriquecimiento opcional
